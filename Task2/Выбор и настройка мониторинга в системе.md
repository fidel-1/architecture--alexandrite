# Выбор и настройка мониторинга в системе

## 1. Мотивация
Внедрение системы технического и бизнес-мониторинга на этом этапе бизнеса критически необходимо компании. Цель - переход от реактивного к проактивному управлению инцидентами. Достижение этой цели приведет к снижению числа инцидентов, сокращению времени на их исправление. Наблюдаемый резкий рост жалоб уже приводит к упущенной выгоде для компании, снижению репутации и снижает ретеншн клиентов. 

**Что это даст бизнесу и компании, детали:**
1. **Снижение оттока B2B-клиентов и упущенной выгоды.** Сейчас Яндекс Метрика показывает только активность B2C-пользователей на фронтенде. Мы абсолютно слепы в отношении API. Мониторинг позволит видеть сбои в заказах API до того, как партнеры расторгнут контракты.
2. **Прозрачность SLA и производительности.** Жалобы на "зависшие на месяцы заказы" и "тормозящие интерфейсы MES" обрабатываются вслепую. Сбор метрик покажет точное время расчета стоимости заказов и реальную скорость работы БД, дав разработчикам понимание "узких горлышек".
3. **Обоснованное масштабирование.** Мониторинг утилизации ресурсов (CPU, RAM, очереди) позволит прогнозировать момент, когда текущая инфраструктура перестанет справляться, и заранее выделять бюджет на новые серверы, избегая аварийных простоев.

--- 

## 2. Выбор подхода к мониторингу
Поскольку система состоит из разных по своей природе компонентов (микросервисы API, брокер сообщений, базы данных), целесообразно использовать гибридный подход:

* **«Четыре золотых сигнала» (Four Golden Signals):** Будем использовать для мониторинга всех программных компонентов (Shop API, CRM API, MES API). Этот подход (Задержка, Трафик, Ошибки, Насыщенность) подходит для веб-сервисов, так как фокусируется на пользовательском опыте + здоровье сервиса.
* **USE (Utilization, Saturation, Errors):** Будем применять для мониторинга инфраструктуры (брокер очередей RabbitMQ, инстансы баз данных Shop DB и MES DB, а также сами EC2-инстансы). Подходит для оценки инфрарструктуры (физических/виртуальных ресурсов).

## 3. Отслеживаемые метрики

### 3.1. Уровень сервисов (API) — Подход: Четыре золотых сигнала
* **Number of requests (RPS) for internet shop API / CRM API / MES API** *(Трафик)*
  * *Зачем нужна:* Показывает общую нагрузку на сервисы. Поможет выявить скачки трафика (особенно от B2B-партнеров) и спланировать автомасштабирование.
  * *Ярлыки:* `endpoint` (URL ручки), `method` (GET/POST), `client_type` (b2b/b2c).
* **Response time (latency) for shop API / CRM API / MES API** *(Задержка)*
  * *Зачем нужна:* Критическая метрика для MES, так как расчет 3D-моделей занимает до 30 минут, а дашборды операторов грузятся медленно. Позволяет контролировать деградацию сервисов.
  * *Ярлыки:* `endpoint`, `percentile` (p50, p95, p99).
* **Number of HTTP 500 / HTTP 200 for shop API / CRM API / MES API** *(Ошибки)*
  * *Зачем нужна:* Для расчета SLA и алертинга при падении сервисов. Метрики 5xx явно укажут на внутренние сбои при обработке заказов.
  * *Ярлыки:* `endpoint`, `status_code`.

### 3.2. Уровень инфраструктуры (БД и RabbitMQ) — Подход: USE
* **Number of dead-letter-exchange (DLX) letters in RabbitMQ** *(Ошибки)*
  * *Зачем нужна:* Ключевая метрика для поиска потерянных заказов. Сообщения, попадающие в DLX, — это заказы, которые CRM или MES не смогли обработать и «бросили».
  * *Ярлыки:* `exchange`, `queue_name`, `error_reason`.
* **Number of messages in flight in RabbitMQ** *(Насыщенность/Утилизация)*
  * *Зачем нужна:* Показывает, сколько сообщений находится в обработке и не подтверждено. Резкий рост означает, что MES/CRM зависли / не справляются (например, из-за долгого расчета 3D-модели).
  * *Ярлыки:* `queue_name`.
* **Memory Utilisation и CPU % (для всех API и DB)** *(Утилизация)*
  * *Зачем нужна:* Базовые метрики здоровья железа. Помогут понять, когда нужно переходить на более мощные инстансы (Scale Up) или добавлять новые (Scale Out).
  * *Ярлыки:* `instance_id`, `service_name`.
* **Number of connections for shop db instance / MES db instance** *(Насыщенность)*
  * *Зачем нужна:* Операторы генерируют огромную нагрузку на БД своими обновлениями. Превышение лимита пула подключений приведет к отказу всей системы.
  * *Ярлыки:* `database_name`, `state` (active/idle).

## 4. План действий (Драфт технического задания)

1. **Развертывание Time-Series БД:** Создать инстанс и установить Prometheus в управляемом контуре для сбора и хранения метрик.
2. **Инструментирование приложений:** * Подключить Spring Boot Actuator и Micrometer в Shop API и CRM API.
   * Интегрировать библиотеку prometheus-net в MES API (C#).
3. **Настройка экспортеров инфраструктуры:**
   * Установить Node Exporter на все EC2 инстансы.
   * Настроить RabbitMQ Prometheus Plugin (встроен в современные версии RabbitMQ).
   * Установить PostgreSQL Exporter для Shop DB и MES DB в Yandex Cloud.
4. **Визуализация:** Развернуть инстанс Grafana, подключить Prometheus как Data Source и собрать 3 дашборда: "Бизнес/Заказы API", "Очереди RabbitMQ" и "Инфраструктура/БД".
5. **Алертинг:** Настроить Alertmanager для интеграции с корпоративным мессенджером / Telegram/Slack/Email по критическим событиям (рост ошибок 5xx, рост DLX).

## 5. Дополнительно: Показатели насыщенности (Saturation) и Алертинг

Для обеспечения стабильности системы выбраны следующие пороговые значения насыщенности:

1. **Метрика:** `Number of connections for MES db instance`
   * *Порог насыщенности:* > 80% от `max_connections` (лимита БД) в течение 2 минут.
   * *Обоснование:* Если пул соединений исчерпается, MES перестанет отвечать вообще.
   * *Действие системы:* Завести тикет с приоритетом `High` на DBA/DevOps. Временное решение: ограничить частоту обновления дашборда на фронтенде MES. Долгосрочное: внедрить кэширование и пулер соединений (PgBouncer).

2. **Метрика:** `Number of messages in flight in RabbitMQ`
   * *Порог насыщенности:* > 500 сообщений в очереди более 5 минут.
   * *Обоснование:* Очередь должна быстро очищаться. Зависание сообщений говорит о том, что MES заблокирован тяжелыми 3D-моделями.
   * *Действие системы:* Отправить алерт в канал техподдержки. Настроить автоматическое масштабирование (Auto-scaling): поднять дополнительный EC2-инстанс MES для разбора очереди.

3. **Метрика:** `CPU % for MES API`
   * *Порог насыщенности:* > 85% в течение 10 минут.
   * *Обоснование:* Длительные вычисления геометрии (полигонов) полностью утилизируют процессор, не оставляя ресурсов на обслуживание других легковесных API-запросов.
   * *Действие системы:* Направить Alert инженерам. Активировать Rate Limiting (ограничение запросов) для внешних B2B API, чтобы защитить систему от полного падения.
