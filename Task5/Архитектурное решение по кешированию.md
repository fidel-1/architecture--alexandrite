# Архитектурное решение по кешированию

## 1. Анализ проблемы и выбор объекта кеширования

На данный момент главным узким горлышком (Bottleneck) системы является база данных `MES db`. Единый инстанс PostgreSQL обслуживает как тяжелые транзакции на запись (новые заказы из RabbitMQ, частые обновления статусов), так и огромный параллельный поток запросов на чтение от операторов. 

Специфика бизнеса создает **паттерн жесткой конкуренции (Race condition)**: вознаграждение оператора напрямую зависит от того, сколько заказов он взял в работу («кто взял заказ, тот и получит оплату»). Это вынуждает всех операторов непрерывно обновлять первую страницу MES (Дашборд), генерируя паразитный DDoS-трафик на чтение. Внедрение пагинации и фильтров не решило проблему производительности, так как база данных тратит ресурсы на сканирование таблиц и расчет `COUNT` при каждом запросе.

**Объект кеширования:** Результаты выборок для дашборда операторов — список доступных для производства заказов (заказы в статусах `MANUFACTURING_APPROVED` и `PRICE_CALCULATED`), с учетом примененных фильтров и пагинации.

---

## 2. Мотивация

Внедрение серверного кеширования преследует следующие цели:
1. **Разгрузка базы данных (DB Offloading):** Перенос 90–95% запросов на чтение из PostgreSQL в оперативную память. СУБД освободит ресурсы CPU и дискового ввода-вывода (IOPS) для критических операций записи новых заказов.
2. **Радикальное снижение задержки (Latency):** Время ответа при чтении из in-memory хранилища составляет единицы миллисекунд (в отличие от десятков секунд при тяжелых SQL-запросах с JOIN-ами). Это решит проблему долгой прогрузки первой страницы MES.
3. **Устранение блокировок:** Интенсивное чтение одних и тех же таблиц больше не будет конкурировать с транзакциями на запись, поступающими из RabbitMQ, что предотвратит зависание статусов заказов.

## 3. Предлагаемое решение

### 3.1. Тип кеширования и технология
Утверждается внедрение **распределенного серверного кеширования (Server-side Distributed Cache)** на базе **Redis**.

* **Почему не клиентское кеширование:** Использование HTTP-кеширования (Cache-Control) или локального хранилища браузера (LocalStorage) категорически запрещено. Так как данные о доступных заказах динамически меняются всеми операторами одновременно, клиентский кеш привел бы к отображению «фантомных» (уже взятых в работу коллег) заказов. Это вызвало бы массовые ошибки валидации на бэкенде и внутренние конфликты среди сотрудников. Система требует единого, разделяемого состояния (Shared State).

### 3.2. Выбор паттерна кеширования

Для интеграции Redis в `MES API` (C#) был проведен сравнительный анализ архитектурных паттернов кеширования:

| Паттерн | Описание | Плюсы решения | Минусы решения | Применимость в Александрите |
| :--- | :--- | :--- | :--- | :--- |
| **Cache-Aside (Lazy Loading)** | Приложение самостоятельно проверяет кеш. Если данных нет (Miss), читает из БД и наполняет кеш. При обновлении данных кеш удаляется. | 1. Максимальная отказоустойчивость: при падении Redis система продолжает работать напрямую через БД.<br>2. Память не забивается лишними данными. | Первый запрос после сброса кеша всегда медленный (Cache Miss Penalty). | **Рекомендовано.** Идеально подходит для систем с высокой долей чтения и критическим требованием к отказоустойчивости инфраструктуры. |
| **Write-Through** | Приложение пишет данные в кеш, а кеш сам синхронно и прозрачно обновляет БД. Чтение идет только из кеша. | Данные в кеше никогда не устаревают. Нулевой риск Cache Miss. | Штраф на скорость записи (Write Penalty) — каждая транзакция ждет ответа от двух систем. | **Не подходит.** Тяжелые расчеты стоимости (до 30 минут) и линейный рост B2B-заказов сделают синхронную двойную запись слишком неэффективной. |
| **Refresh-Ahead** | Кеш автоматически обновляет свои значения по таймеру до того, как истечет их TTL, на основе аналитики частоты запросов. | Минимизация задержек для пользователя, кеш всегда "горячий". | Сложная реализация. Высокий риск паразитной нагрузки на БД для прогрева редко запрашиваемых фильтров. | **Не подходит.** Поступление новых заказов имеет случайный характер (зависит от внешних B2B-продавцов), паттерн неэффективен. |

**Итог:** Выбран паттерн **Cache-Aside** как наиболее надежный, простой в реализации разработчиком на C# и гарантирующий работоспособность производства даже в случае аварии на узле Redis.

### 3.3. Диаграмма последовательности (Sequence Diagram)

Процесс взаимодействия компонентов при чтении списка заказов (сценарий Cache Miss / Cache Hit) и при изменении статуса заказа (инвалидация) выглядит следующим образом:

```mermaid
sequenceDiagram
    autonumber
    actor Operator as Оператор MES
    participant MES_UI as MES Frontend (React)
    participant MES_API as MES API (C#)
    participant Redis as Redis Cache
    participant DB as MES DB (PostgreSQL)

    rect rgb(240, 248, 255)
    Note over Operator, DB: Сценарий Чтения: Запрос дашборда со списком заказов
    Operator->>MES_UI: Открывает первую страницу / применяет фильтр
    MES_UI->>MES_API: GET /api/v1/orders?status=APPROVED&page=1
    MES_API->>Redis: GET orders:list:approved:p1
    
    alt Данные найдены (Cache Hit)
        Redis-->>MES_API: Возвращает закешированный JSON (список заказов)
    else Данные не найдены (Cache Miss)
        Redis-->>MES_API: null / expired
        MES_API->>DB: SELECT * FROM orders WHERE status='APPROVED' LIMIT 10
        DB-->>MES_API: Результат выборки из PostgreSQL
        MES_API->>Redis: SET orders:list:approved:p1 [Данные] + TTL (5 min)
    end
    
    MES_API-->>MES_UI: 200 OK + JSON (Список заказов)
    MES_UI-->>Operator: Отображает актуальный дашборд
    end

    rect rgb(255, 240, 245)
    Note over Operator, DB: Сценарий Записи: Изменение статуса (Оператор взял заказ)
    Operator->>MES_UI: Нажимает кнопку "Взять в работу заказ #777"
    MES_UI->>MES_API: POST /api/v1/orders/777/accept
    MES_API->>DB: UPDATE orders SET status='MANUFACTURING_STARTED' WHERE id=777
    DB-->>MES_API: Транзакция успешна
    
    Note over MES_API, Redis: Программная инвалидация кеша (Сброс)
    MES_API->>Redis: DEL orders:list:approved:* (Очистка всех страниц списков этого статуса)
    Redis-->>MES_API: OK
    
    MES_API-->>MES_UI: 200 OK (Заказ успешно закреплен за вами)
    MES_UI-->>Operator: Интерфейс обновляется
    end
